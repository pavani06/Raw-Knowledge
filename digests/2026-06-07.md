# Digest 2026-06-07

Two talks from the AI Engineer conference, both recorded in 2026, both about the same
problem from complementary angles: **how do you build software when agents do the
implementation?** Together they form the founding cluster of this ontology — 23 concepts,
8 entities, and a coherent thesis about the emerging discipline of agent-era engineering.

---

## Sources Covered

### 1. Anthropic Workshop: Build Agents That Run for Hours
[[sources/2026-06-07-anthropic-workshop-build-agents-that-run-for-hours|Anthropic Workshop: Build Agents That Run for Hours]]
— Ash Prabaker & Andrew Wilson (AI Engineer conf, 2026-05-18, 1h 15m)

A history tour of the past year of Claude / Claude Code releases, followed by the
current state-of-the-art harness pattern. Core thesis: **long-horizon autonomous agents
are made possible not by the model alone but by an evolving [[concepts/agent-harness]]**.
Today's best pattern is an adversarial [[concepts/generator-evaluator-pattern]] with
separate context windows, file-based state, and a planner that negotiates a "contract"
of done-ness before any code is written. The talk is greenfield-oriented and explicitly
names observability for ultra-long runs as unsolved.

### 2. Harness Engineering: How to Build Software When Humans Steer, Agents Execute
[[sources/2026-06-07-harness-engineering-how-to-build-software-when-humans-steer-agent|Harness Engineering: How to Build Software When Humans Steer, Agents Execute]]
— Ryan Lopopolo, OpenAI (AI Engineer conf, 2026-04-17, 46m)

A practitioner's keynote from someone who has built software exclusively with agents for
nine months, spending over a billion output tokens per day. Core thesis: **implementation
is no longer the scarce resource** — [[concepts/code-as-free-resource|code is free]].
The scarce resources are human time, human/model attention, and context window. The
engineer's role shifts to defining work, writing guardrails, and building self-improving
feedback loops. Ryan names this discipline [[concepts/harness-engineering]].

## Emerging Themes

### 1. The harness is the product — and it is disposable

Both sources converge on the same structural insight from different directions.

Anthropic's team shows the *technical* co-evolution: nearly every model release over the
past year shipped alongside harness changes. Sprint decomposition and fresh context
windows were critical for Opus 4.5; both were dropped for Opus 4.6 once
[[concepts/compaction]] made long coherent sessions viable. *"Our harness wasn't wrong —
it was right for the previous model; the frontier moved."*

Ryan's team shows the *operational* co-evolution: the harness is a set of guardrails,
lint rules, docs, and [[concepts/reviewer-agents]] that encode what good code looks like.
As the model improves, some guardrails become unnecessary; others need to be added for
new failure modes. The harness is never finished — it is a living artifact.

> [!inference]
> The implication across both sources: **build harnesses that are easy to simplify**.
> Every component should be deletable. The engineering discipline is not "build the
> perfect harness" but "build the harness that is right for today's model, and know which
> parts to delete next." This is a fundamentally different design philosophy than
> traditional software infrastructure, which is built to last.

### 2. Adversarial separation beats self-reflection — at every layer

The Anthropic talk makes this explicit at the architecture level: the
[[concepts/generator-evaluator-pattern]] works because tuning a harsh standalone critic
is *tractable*, while tuning a builder to be self-critical is *not*. Self-evaluation is
called "a trap." The [[concepts/planner-generator-evaluator-architecture]] extends this
by adding a planner that negotiates a contract of done-ness before generation begins —
preventing the evaluator from moving the goalposts.

Ryan's talk applies the same principle at the team/process level: [[concepts/reviewer-agents]]
are persona-based sub-agents (front-end architect, reliability engineer, security
reviewer) that trigger on every push and inject feedback into the PR. The implementation
agent can acknowledge, defer, or reject feedback — but it cannot ignore it. The human
is removed from the synchronous review loop entirely.

Both sources are describing the same architectural pattern at different scales:
**separate the critic from the creator, give each its own context, and let them
negotiate**. The Anthropic version is a single-session adversarial loop; Ryan's version
is a CI-integrated continuous loop across the entire development lifecycle.

### 3. Context is the master constraint — and the harness is its manager

[[concepts/context-window-management]] is the umbrella over a surprising number of
innovations in both talks. The Anthropic source names the failure mode:
[[concepts/context-rot]] / context anxiety. The mitigations it describes —
[[concepts/compaction]], [[concepts/skills-progressive-disclosure]],
[[concepts/programmatic-tool-calling]], [[concepts/file-system-state]] — are all ways
to keep the model effective within a finite window.

Ryan's source adds a different angle: **just-in-time context surfacing**. Don't
frontload all instructions (which overwhelms the agent); defer them to lint/test time
so the agent can prototype freely, then receive targeted corrections at the moment they
are relevant. [[concepts/codebase-uniformity]] is framed as a context-management
strategy — one way to do everything means the model builds transferable context
regardless of where in the repo it is working.

The two approaches are complementary: Anthropic's is about *managing what's in the
window*; Ryan's is about *controlling when things enter the window*. Together they
describe a complete context discipline.

### 4. Verification means *using* the thing — at every stage

Both sources insist that verification must go beyond static analysis and unit tests.

The Anthropic talk's [[concepts/verification-loop]] is the evaluator driving a live app
with [[entities/playwright]], catching bugs (FastAPI route ordering, delete-key boolean
logic) that CI and unit tests miss. *"You can grade taste if you write down a strong
opinion"* — the [[concepts/design-taste-rubric]] extends this to subjective quality.

Ryan's talk extends verification across the entire development lifecycle: QA plans
document critical user journeys and must be attached to every user-facing PR; reviewer
agents validate the plan is present and complete; smoke-testing agents download and
launch built artifacts before promotion to distribution. Verification is not a step —
it is a continuous, automated, multi-stage process.

> [!inference]
> The shared principle: **verification is only meaningful if it exercises the actual
> artifact in its actual environment**. Static analysis and unit tests are necessary but
> not sufficient. The agent must use the app, read the UI, check the network errors,
> validate the user journeys. This is the same insight that drove the shift from unit
> tests to integration tests in traditional software — but applied to agent output.

### 5. The human skill is specification, not implementation

Both sources describe the same role shift for human engineers, but frame it differently.

Anthropic frames it as **reading traces**: the primary harness-debugging discipline is
hand-reading agent traces and empathizing with the model — finding where its judgment
diverged from a human's, then tuning the prompt. The human's value is in knowing what
good looks like and being able to identify where the agent went wrong.

Ryan frames it as **writing down non-functional requirements**: agents have seen
trillions of lines of code and can satisfy any style, but only if you specify it.
Undocumented taste is invisible to the model. The engineer's job is to make the quality
bar legible — through ADRs, persona-oriented docs, lint rules, and
[[concepts/garbage-collection-day]] rituals that convert observed failures into durable
guardrails.

Both framings point to the same underlying shift: **the engineer's primary output is
no longer code but specification** — the prompts, guardrails, rubrics, and process
artifacts that shape what the agents produce.

## Contradictions & Evolving Understanding

> [!contradiction] Compaction vs. coherence (internal to Source 1)
> The Anthropic talk embraces single-session + [[concepts/compaction]] (replacing fresh
> context windows) yet warns "compaction ≠ coherence — lossy summaries drift." The
> resolution offered is model- and use-case-dependent: Opus 4.6 holds coherence well
> enough for most work, but harder tasks still benefit from structured hand-offs.
> Ryan's source implicitly supports the structured-hand-off side: his harness uses
> reviewer agents and lint failures as continuous context refreshers precisely because
> auto-compaction pages out context over time. **Watch for future sources that take a
> firm side on whether compaction is sufficient or structured hand-offs remain necessary.**

> [!contradiction] Ralph Loop — dead or still useful?
> [[concepts/ralph-loop]] is positioned in Source 1 as superseded (no contract
> negotiation, patches instead of pivots), but an audience "smart-zone/dumb-zone"
> argument defends task-slicing to keep the model sharp. The speakers concede it "still
> has its place depending on use case." Ryan's source never mentions it — his harness
> skips plan mode entirely ("I'm never going to read it anyway") and relies on tickets
> + skills + reviewer agents instead. The two sources together suggest the Ralph Loop
> is a useful primitive for simple tasks but insufficient for complex, long-horizon work.

> [!contradiction] Greenfield vs. brownfield applicability
> Source 1 explicitly flags that the generator-evaluator pattern is greenfield-oriented
> and that brownfield/production use needs more control and custom rubrics. Source 2
> describes a team that has been running this methodology on a production codebase for
> nine months. The difference may be that Ryan's approach (harness engineering as
> operational discipline) is more brownfield-compatible than Anthropic's (adversarial
> generator-evaluator as a single-session pattern). **This tension is unresolved and
> worth tracking as more production case studies emerge.**

## Writing / Research Prompts

1. **"The specification economy"** — Both sources argue that the engineer's primary
   output is shifting from code to specification (prompts, guardrails, rubrics, docs).
   Write up what this means for engineering team structure, hiring, and career
   development. What skills become more valuable? What becomes obsolete? Pull from
   [[concepts/non-functional-requirements]], [[concepts/harness-engineering]], and
   [[concepts/reading-traces]].

2. **"Adversarial separation as a universal pattern"** — The
   [[concepts/generator-evaluator-pattern]] (Anthropic) and [[concepts/reviewer-agents]]
   (Ryan) are both instances of the same principle: separate the critic from the creator,
   give each its own context, let them negotiate. Trace this pattern across both sources
   and argue whether it is a fundamental architectural principle of agent systems or a
   temporary workaround for models that can't self-evaluate reliably. Connect to the
   GAN analogy and the sycophancy failure mode.

3. **"Context engineering as the new systems programming"** — Both sources treat context
   management as the central engineering discipline of the agent era. Map the full
   taxonomy of context-management techniques across both sources:
   [[concepts/compaction]], [[concepts/skills-progressive-disclosure]],
   [[concepts/programmatic-tool-calling]], [[concepts/file-system-state]],
   just-in-time surfacing, [[concepts/codebase-uniformity]], file size limits, reviewer
   agents as context refreshers. Argue which of these are durable (won't be obsoleted
   by model improvements) and which are temporary scaffolding.
