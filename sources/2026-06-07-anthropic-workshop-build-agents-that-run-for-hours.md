## Key Takeaways

- **Long-running agents need a harness, not just a better model.** The model and the [[concepts/agent-harness]] co-evolve; capability rose from ~1h (Sonnet 3.7) to ~12h (Opus 4.6) on a minimal-scaffold benchmark over a year ([[concepts/long-running-agents]]).
- **Self-evaluation is a trap; use an adversarial evaluator.** A separate critic with its own context window is far more reliable than asking a builder to grade itself ([[concepts/generator-evaluator-pattern]]).
- **Tuning a harsh standalone critic is tractable; tuning a self-critical builder is not** — exploit the gap between an LLM's ability to critique vs. generate.
- **The evaluator must actually use the app** ([[entities/playwright]]) — driving the UI catches bugs (route ordering, key handlers) that CI and unit tests miss ([[concepts/verification-loop]]).
- **Generator + evaluator negotiate a "contract" of what done means** (27 criteria in one app) before any code is written — the key innovation over the fixed `plan.md` of the [[concepts/ralph-loop]] ([[concepts/planner-generator-evaluator-architecture]]).
- **Keep the planner high-level** so early errors don't cascade across sprints; keep evaluator/generator context streams clean (critic judges output only, not the generator's reasoning).
- **Use the file system for shared state**, preferring JSON (models overwrite it less than markdown) and leaving timestamped "breadcrumbs" for resumption ([[concepts/file-system-state]]).
- **Harness design is model-specific and disposable.** What was critical for Opus 4.5 (sprint decomposition, fresh context windows) was dropped for Opus 4.6 + [[concepts/compaction]]. The harness wasn't "wrong" — the frontier moved.
- **Compaction ≠ coherence** — lossy summaries drift; structured hand-offs remain valuable.
- **The real debugging loop is reading traces by hand** and empathizing with the model — the same muscle as reading a stack trace ([[concepts/reading-traces]]).
- **You can grade taste** if you write down a strong opinion (design/originality/craft/functionality rubric, few-shot calibrated) ([[concepts/design-taste-rubric]]).
- **This pattern is greenfield-oriented** — brownfield/production use needs more control and custom rubrics; observability for ultra-long runs is unsolved.
